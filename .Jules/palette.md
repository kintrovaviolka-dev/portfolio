## 2026-06-13 - Keyboard Accessibility for Hover-Driven State Changes
**Learning:** When interactive elements rely on structural or visual transformations (like flip cards) that occur on hover, blind mapping of hover states to focus states causes problems. Users cannot control the state change via keyboard activation (Enter/Space), which leads to frustrating and inaccessible experiences.
**Action:** When making custom non-interactive elements (like flip cards with `onclick`) keyboard accessible, explicitly add `tabindex="0"`, `role="button"`, an `aria-label`, and an `onkeydown` handler to support `Enter` and `Space` key activation. Provide clear `:focus-visible` outlines, but do NOT map `:hover` states directly to `:focus-visible` if it triggers transformations that should be user-controlled.
