## 2025-03-01 - Interactive Element Keyboard Accessibility & Focus
**Learning:** Adding `tabindex="0"` to make non-standard elements (like `div` or `article` with `onclick` handlers) focusable is not enough for accessibility. We must also define `role="button"` and handle keyboard activation (`Enter` and `Space`) via an `onkeydown` event handler, taking care to `preventDefault` for `Space` so it doesn't cause page scrolling. Additionally, blind mapping of `:hover` effects to `:focus-visible` can be problematic if the hover state triggers visual/structural changes (like a 3D flip) which should ideally only be toggled upon explicit activation rather than just focus.
**Action:** When making custom interactive elements accessible, always provide full keyboard support (`role`, `tabindex`, explicit activation handler). For state-changing animations like card flips, ensure the structural visual shift relies on the interaction state (e.g. adding a class like `.flipped` on click/keypress) rather than triggering immediately on focus, but provide a distinct `:focus-visible` outline so users know the element is targeted.
