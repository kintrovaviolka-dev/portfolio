## 2024-06-12 - Making Hover-Triggered Structural Transformations Keyboard Accessible
**Learning:** When dealing with custom non-interactive elements (like `<article>`) that undergo visual/structural transformations on `:hover` (e.g., CSS 3D card flips), blindly mapping `:hover` states to `:focus-visible` is an anti-pattern. This approach forces the state change upon focus, depriving keyboard users of manual activation control.
**Action:** Instead of triggering the transformation on focus, I applied a standard outline via `:focus-visible` for navigation visibility. I then added `tabindex="0"`, `role="button"`, `aria-label`, and an explicit `onkeydown` handler supporting `Enter` and `Space` (using `event.preventDefault()` for Space to prevent page scroll) to allow keyboard users to deliberately trigger the state change, mirroring the manual intent of a mouse click.
